Data plane

AKS hosts the RAG and data plane.

The retrieval path is explicit: application requests land in AKS, data is fetched from the durable backends, and the assistant returns grounded results without mixing the worker tier into the service boundary.

Service path

  • AKS carries the RAG and data plane services.
  • pgvector is the custom retrieval integration path.
  • PostgreSQL stores durable state and retrieval metadata.
  • Azure NetApp Files supports the shared file substrate.

Request flow

Requests move through a grounded path instead of a monolithic assistant.

The site explains the data plane in terms of deterministic service steps so the public model can distinguish inference, retrieval, and storage access.

Ingress

Requests arrive at the AKS-hosted service boundary and are checked before retrieval starts.

Retrieval

pgvector powers the semantic lookup path as a custom integration, not a generic managed feature.

Grounding

PostgreSQL and the shared storage layer provide stable references for the response assembly process.

Return path

The response is emitted from the data plane while the worker tier stays outside the retrieval service boundary.

Custom integration

pgvector is a deliberate engineering path.

The architecture does not pretend that vector retrieval is a drop-in checkbox. pgvector is called out as a custom integration path so the implementation work is visible and reviewable.

Indexing

The vector index is managed alongside PostgreSQL state so retrieval and metadata stay co-located.

Operational control

Service behavior is kept in AKS, which makes the data plane easier to scale and observe separately from the worker tier.
