Storage

Azure NetApp Files anchors the durable workspace.

The storage story keeps file-backed collaboration, PostgreSQL state, and retrieval index data visible as separate concerns. That helps explain how the architecture can be stable without pretending the assistant is stateless.

Persistence model

  • Azure NetApp Files supports the shared file layer.
  • PostgreSQL preserves metadata and delivery state.
  • pgvector is the custom integration path for retrieval indexes.
  • Published artifacts stay in a repo-owned store with stable filenames.

Durability

Each persistence layer has a specific role.

The architecture keeps file storage, relational state, and vector retrieval separate so the public story is easy to audit and the implementation can evolve without collapsing those roles together.

Azure NetApp Files

Shared storage for the working surface, collaboration paths, and artifact handoff.

PostgreSQL

System-of-record storage for metadata, workflow state, and delivery evidence.

pgvector

The vector path is custom engineered so retrieval behavior is observable and intentional.

Artifact store

The public PNG boards and PDF stay in a visible repo-owned location with stable filenames.

Artifact paths

The published files are linked directly from the pages.

The public site references the boards and deck by their stable filenames so the artifact set can be reviewed without guessing which copy is current.
