Architecture

Hybrid baseline, split control planes, and a visible worker boundary.

The supported baseline is hybrid. AKS hosts the RAG and data plane, while the NemoClaw/OpenShell worker tier stays separate so the public architecture can show where execution, retrieval, and storage are intentionally divided.

Baseline summary

  • Supported baseline is hybrid, not a single-zone public cloud-only diagram.
  • AKS hosts the RAG and data plane for retrieval and service orchestration.
  • NemoClaw/OpenShell worker tier remains separate from the retrieval plane.
  • pgvector is a custom integration path, not a generic checkbox feature.

Layering

The architecture is organized around explicit trust and runtime boundaries.

The public pages keep the control plane, data plane, and worker tier legible. That makes it easier to explain where the assistant can retrieve, where it can execute, and where policy gates the transition between the two.

Control plane

NemoClaw coordinates onboarding, policy, and delivery orchestration but does not collapse the worker boundary.

AKS data plane

AKS hosts the retrieval and service tier so RAG behavior stays isolated from the worker runtime.

Worker tier

NemoClaw/OpenShell workers remain separate to preserve the intended trust split and sandbox control.

Storage and state

Azure NetApp Files and PostgreSQL hold the durable project state that the retrieval plane consumes.

Baseline

The hybrid model is the supported baseline.

The incubator baseline assumes a hybrid deployment, with enterprise concerns expressed through concrete storage, policy, and retrieval boundaries rather than an abstract all-in-one platform.

Enterprise side

Holds the persistent services, data access rules, and storage-backed state used by the public reference stack.

Sandbox side

Runs the worker tier that interprets instructions, prepares output, and hands off to the data plane by policy.
